Write-up notes

Click on the firewall next to Chris Elgee to open the Holiday Firewall Simulator. This interface lets us configure rules between the Internet, DMZ, Internal Network, Workstations, and Cloud Services to meet the listed security goals.

Holiday Firewall Simulator start screen showing the network zones and firewall configuration goals we must satisfy.

Holiday Firewall Simulator network map showing Internet, DMZ, Internal Network, Workstations, and Cloud Services, along with the service port legend.

Goal 1: Internet → DMZ – Allow only HTTP and HTTPS traffic

Write-up notes

Our first firewall configuration goal is: Internet to DMZ: Allow only HTTP and HTTPS traffic.

From the main Holiday Firewall Simulator screen, click the Internet node on the network map to open the Internet → DMZ connection editor.

  1. In the Connection to DMZ (Demilitarized Zone) section, enable only:
    • HTTP (Port 80)
    • HTTPS (Port 443)
    Leave DNS, SMTP, SMB, and SSH unchecked.
  2. Click Save Rules to apply the configuration.

The screenshot below shows the correct settings for this goal, with only HTTP and HTTPS allowed from the Internet to the DMZ.

Internet → DMZ rules configured to allow only HTTP (80) and HTTPS (443) from the Internet into the DMZ.

Goal 2: DMZ → Internal – Allow HTTP, HTTPS, and SSH traffic

Write-up notes

The next firewall requirement is: DMZ to Internal: Allow HTTP, HTTPS, and SSH traffic.

From the network map, click the DMZ node to open the DMZ configuration dialog. You’ll see two connections listed: Connection to Internal Network and Connection to Internet. The Internet connection is already correctly configured (shown in green), but the Connection to Internal Network still needs work.

  1. Click Connection to Internal Network to open its rule editor.
  2. In the list of protocols, enable only:
    • HTTP (Port 80)
    • HTTPS (Port 443)
    • SSH (Port 22)
    Leave DNS, SMTP, and SMB unchecked so that only web and SSH traffic can flow from the DMZ into the Internal Network.
  3. Click Save Rules to apply the configuration. The DMZ → Internal goal should now be marked as Completed on the Firewall Configuration Goals list.

The screenshot below shows the DMZ rules after applying the correct configuration for this goal.

DMZ → Internal Network rules with HTTP (80), HTTPS (443), and SSH (22) allowed, satisfying the “DMZ to Internal” configuration goal.

Goal 3: Internal → DMZ – Allow HTTP, HTTPS, and SSH traffic

Write-up notes

When we click the Internal Network node on the map, the configuration console shows the Connection to the DMZ (Demilitarized Zone) is already correctly configured and marked in green. That means the Internal to DMZ goal is satisfied with HTTP, HTTPS, and SSH already allowed, so we don’t need to change anything for this rule.

However, we can also see that the Connection to Cloud Services and Connection to Workstations entries are still red and will be addressed in the remaining goals.

Internal Network view: the DMZ connection is complete (green), while the Cloud Services and Workstations connections still need rules.

Goal 4: Internal → Cloud – Allow HTTP, HTTPS, SSH, and SMTP traffic

Write-up notes

For the next requirement, we need: Internal to Cloud: Allow HTTP, HTTPS, SSH, and SMTP traffic.

  1. From the network map, click the Internal Network icon.
  2. In the dialog, choose Connection to Cloud Services to open its rule editor.
  3. In the list of protocols, enable exactly:
    • HTTP (Port 80)
    • HTTPS (Port 443)
    • SSH (Port 22)
    • SMTP (Port 25)
    Leave DNS and SMB unchecked so only the required web, SSH, and mail traffic can reach the cloud services from the internal network.
  4. Click Save Rules. Back on the Firewall Configuration Goals list, the Internal to Cloud goal should now show as Completed.

At this point, all cloud-bound traffic that users need (web, secure web, SSH admin access, and outbound mail) is allowed, while unnecessary services remain blocked.

Internal → Cloud Services connection configured with HTTP, HTTPS, SSH, and SMTP allowed, matching the goal requirements.

Goal 5: Internal → Workstations – Allow all traffic types

Write-up notes

The final rule is: Internal to Workstations: Allow all traffic types. Workstations are on the trusted internal side of the network, so this lab models them as being allowed to receive any protocol from the internal network segment.

  1. Click the Internal Network icon on the network map again.
  2. Select Connection to Workstations to open the rule editor.
  3. Enable all available options in the list: HTTP, HTTPS, SSH, SMB, SMTP, and DNS.
  4. Click Save Rules. The connection indicator turns green, and the Internal to Workstations goal is marked Completed in the Firewall Configuration Goals panel.

With this change, the internal network can reach all necessary services on the workstation segment, while earlier goals ensured that direct Internet access to the internal network is still blocked.

Internal → Workstations connection with all protocols enabled, completing the final traffic rule.
