Working through the Challenge:
Write-up notes

Click on the Cranberry Pi terminal, we are presented with an interactive networking visualization tool. The challenges are pretty straightforward; we just need to configure and submit to solve each challenge.

Challenge 1: DNS Lookup

Write-up notes

For Challenge 1, we use DNS to find the IP address of visual-networking.holidayhackchallenge.com. Set the port to 53, the domain name to visual-networking.holidayhackchallenge.com, and the request type to A, then submit the DNS request to see the IPv4 address returned.

Challenge 1 – DNS Lookup solution showing a successful A record resolution for visual-networking.holidayhackchallenge.com.

Challenge 2: TCP 3-Way Handshake

Write-up notes

In this step, we build a classic TCP 3-way handshake between the client and the server so they can establish a reliable connection.

First, configure the client → server packet with the SYN flag set. Next, configure the server → client response with SYN and ACK set. Finally, the third client → server packet with the ACK flag.

Once the packets are in the SYN → SYN/ACK → ACK order challenge 3 will unlock.

Challenge 2 – TCP 3-Way Handshake solution showing the SYN → SYN/ACK → ACK exchange between client and server.

Challenge 3: HTTP GET Request

Write-up notes

For this step, we use our newly established TCP connection to send a simple HTTP GET request and retrieve the web page.

On the Client side, configure the request as follows:

  • HTTP Verb: GET
  • HTTP Version: HTTP/1.0
  • Host: visual-networking.holidayhackchallenge.com
  • User-Agent: leave blank (not required for this step)

After clicking Send HTTP Request, the server responds with an HTTP 302 Found redirect to the HTTPS version of the site. When you see the success message and the green “✔ HTTP Challenge Complete!” text, Challenge 3 is solved.

Challenge 3 – HTTP GET Request solution showing a valid GET / HTTP/1.0 request to visual-networking.holidayhackchallenge.com and the 302 redirect response.

Challenge 4: TLS Handshake

Write-up notes

Now that we’ve sent a basic HTTP request, the next step is to secure the connection with TLS. In this challenge, we drag and drop the TLS messages into the correct order to build a proper handshake between the client and the secure web server.

We’re connecting to https://visual-networking.holidayhackchallenge.com. The correct sequence of TLS messages is:

  1. Client Hello → the client proposes supported ciphers and parameters
  2. ← Server Hello the server selects the cipher suite
  3. ← Certificate the server sends its SSL/TLS certificate
  4. Client Key Exchange → the client sends key material used to derive the shared secret
  5. ← Server Change Cipher Spec the server indicates that subsequent messages are encrypted
  6. ← Finished the server confirms the handshake is complete

Once the messages are in this order, the interface shows “✔ TLS Handshake Complete!”, meaning our HTTP traffic can now travel inside an encrypted TLS tunnel.

Challenge 4 – TLS Handshake solution showing the correct sequence of TLS messages and a completed secure connection.

Challenge 5: HTTPS GET Request

Write-up notes

With the TLS tunnel established in Challenge 4, we can now send an HTTPS request over the secure connection. For this step, we keep things very similar to the earlier HTTP challenge, but make sure we’re talking to the HTTPS service instead.

In the Client panel, set:

  • HTTP Verb: GET
  • HTTP Version: HTTP/1.0
  • Host: visual-networking.holidayhackchallenge.com
  • User-Agent: leave blank (the challenge doesn’t require a value)

Then click Send HTTPS Request. A valid 200-style response is returned on the right, and the status box at the bottom reports that the HTTPS Challenge is complete. At this point we’ve walked the full path: DNS → TCP → HTTP → TLS → HTTPS.

Challenge 5 – HTTPS GET Request solution showing a successful secure HTTPS request to visual-networking.holidayhackchallenge.com over TLS.
